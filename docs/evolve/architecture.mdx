## Overview

Kapso uses a modular architecture with pluggable components connected through factories. The system is designed around a central orchestration loop that coordinates experimentation.

## Component Responsibilities

### Evidence-directed Generic path

```mermaid
flowchart LR
    O["Orchestrator: budget and fidelity"] --> E["Evidence + deterministic policy"]
    E --> A["IdeaArchive: PLANNED"]
    A --> G["Codex / Claude candidate calls"]
    G --> N["Rules + OpenAI embedding neighbors"]
    N --> S["Codex / Claude selector"]
    S --> B["IdeaArchive: SELECTED / BRIDGED"]
    B --> X["Implementation + evaluation (registered when configured)"]
    X --> W["Evidence author"]
    W --> V["Optional external evaluator"]
    V --> H["ExperimentHistoryStore"]
    H --> C["IdeaOutcome"]
    C --> K["Run checkpoint"]
```

The candidate population, analyses, and decision are durable before execution.
Executed memory is then written before the linked idea outcome, and the run
checkpoint is last. See the
[implemented-system reference](../research/ideation-v3-implemented-system.md)
for the exact lifecycle and recovery seams.

### Kapso (Main API)

The user-facing entry point that provides the four-pillar API:

```python
class Kapso:
    def research(objective, mode, depth) -> Source.Research
    def learn(*sources, wiki_dir) -> PipelineResult
    def evolve(goal, ...) -> SolutionResult
    def deploy(solution, strategy) -> Software
    def index_kg(wiki_dir, save_to) -> str
```

### OrchestratorAgent

The central coordinator that manages the solve loop:

```python
def solve(self, experiment_max_iter, time_budget_minutes, cost_budget):
    register_and_reconcile_evaluation()
    save_bootstrap_checkpoint()
    for i in range(experiment_max_iter):
        capacity = build_budget_and_fidelity_snapshot()
        if capacity.exhausted:
            save_resumable_budget_stop()
            break
        node = self.search_strategy.run(problem, capacity.progress_percent)
        finalized = new_nodes_or_execution_revisions(node)
        attach_optional_external_evaluation(finalized)
        persist_experiment_then_idea_outcome(finalized)
        save_checkpoint(completed=node.should_stop)
```

### Pluggable Components

All major components are created via factories and can be swapped via configuration:

| Component | Factory | Registered Types |
|-----------|---------|------------------|
| Search Strategy | `SearchStrategyFactory` | `generic`, `benchmark_tree_search` |
| Knowledge Search | `KnowledgeSearchFactory` | `kg_graph_search`, `kg_llm_navigation` |
| Coding Agent | `CodingAgentFactory` | `aider`, `gemini`, `claude_code`, `openhands` |
| Feedback Generator | Uses `CodingAgentFactory` | Same as coding agents (default: `claude_code`) |

## Data Flow

1. **Problem Handler** provides the complete problem.
2. **GenericSearch** projects linked `SearchNode` history into frozen evidence.
3. **Policy and operators** choose bootstrap/explore/exploit/verify/recover and
   concrete parent plans.
4. **Candidate agents, analyzer, embeddings, and selector** advance the
   `IdeaArchive` through `PLANNED → GENERATED → ANALYZED → SELECTED`.
5. **Experiment Workspace** freezes the parent ref and creates the linked
   branch/node (`BRIDGED`).
6. **Implementation, integrity, registered evaluation, feedback, and evidence
   authoring** finalize the node.
7. **Orchestrator** attaches optional external evaluation, writes the executed
   `ExperimentRecord`, writes the `IdeaOutcome`, then checkpoints.
8. **RepoMemory** remains branch-specific repository understanding.

## Key Design Patterns

### Factory Pattern

All pluggable components use factories with decorator-based registration:

```python
# Registration
@register_strategy("generic")
class GenericSearch(SearchStrategy):
    ...

# Creation
strategy = SearchStrategyFactory.create(
    strategy_type="generic",
    problem_handler=handler,
    llm=llm,
    coding_agent_config=config,
    params=params,
)
```

### Configuration Modes

Configuration is organized into modes that bundle related settings:

```yaml
modes:
  GENERIC:
    ideation_profile: "DEFAULT"
    search_strategy:
      type: "generic"
      params: { ... }
    coding_agent:
      type: "aider"
      model: "gpt-4o-mini"
    context_manager:
      type: "kg_enriched"
    knowledge_search:
      type: "kg_graph_search"
      enabled: true
```

### Git-Based Experiment Isolation

Each experiment runs on its own git branch, enabling:
- Parallel experimentation
- Easy rollback to any state
- Tree-based solution exploration
- RepoMemory inheritance across branches
