---
title: "Ideation Parent Plans"
description: "How evidence-directed operators choose and freeze experiment lineage"
---

Generic search has no campaign-wide `parent_policy`. The deterministic policy
first chooses a campaign mode; operator planning then assigns each candidate
one explicit parent plan.

| Parent plan | Meaning |
|---|---|
| `baseline` | Build from `main` |
| `best_valid` | Build from the strongest comparable experiment under the registered evaluator/fidelity/fraction/seed identity |
| `specific_experiment` | Build from the named earlier node |
| `recover_branch` | Continue the named recoverable experiment on the same node |

Exploration operators can start from the baseline or a specific experiment;
exploitation operators usually refine the best valid result; recovery is only
legal for an interrupted linked execution. The plan comes from evidence and
operator intent, not from iteration number or an LLM-selected branch string.

## Frozen resolution

Before candidate generation, Kapso resolves every plan into one immutable
snapshot:

```text
ResolvedParentSnapshot
  node_id
  branch_name
  git_ref
  materialized_ref
  diff_base_ref
  feedback_base_ref
```

The generator reads the exact `materialized_ref`. If that candidate is selected,
the same snapshot supplies implementation ancestry, `SearchNode.parent_node_id`,
the code-diff base, and the feedback base. Kapso checks the branch still points
to the frozen Git ref immediately before implementation and fails if it moved.
There is no fallback to `main`.

Candidates sharing a ref reuse one read-only materialization. Candidates with
different plans see different repository states. A crossover may cite several
ideas or experiments, but it still has exactly one implementation parent and
therefore unambiguous Git ancestry.

## Persistence and resume

The resolved snapshots are part of the durably persisted `IdeaBatch`, and the
selected snapshot is copied to the selected `IdeaRecord` and linked
`SearchNode`. Resume resolves the same plans again and requires exact equality
with the persisted snapshots. A missing ref, moved branch, changed problem, or
changed frozen planning context fails loudly; Kapso never silently chooses a
new parent or creates a replacement batch.

`recover_branch` preserves the original idea ID, experiment node ID, branch,
and parent snapshot while incrementing the node's execution revision. This
keeps recovery distinct from a new hypothesis.
